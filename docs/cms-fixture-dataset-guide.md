# CMS Fixture Dataset Guide

## Purpose

This repository contains a deterministic, fictional dataset for testing the CMS,
public API, website listings, filters, detail pages, homepage aggregation, media,
documents, enquiries, and dashboard reports.

The workflow is guarded: it refuses to operate unless the target database name
ends in `_seed`. It does not modify `sidhkofed_cms`.

## Connection

Default fixture connection:

```text
postgresql://postgres:sa3421@127.0.0.1:5432/sidhkofed_seed?schema=public
```

Override it with `SEED_DATABASE_URL`; the database name must still end in
`_seed`. PostgreSQL 18 is expected under `C:\Program Files\PostgreSQL\18\bin`.
Set `POSTGRES_BIN` when it is installed elsewhere.

The only CMS login is the existing Super Admin from:

- `SEED_SUPERADMIN_EMAIL`
- `SEED_SUPERADMIN_PASSWORD`
- `SEED_SUPERADMIN_NAME`

## Create Or Reset

```powershell
npm run db:fixtures:setup
npm run db:fixtures:verify
```

`setup` creates the isolated database when needed, applies migrations, and seeds
the fixtures. To discard and rebuild it:

```powershell
npm run db:fixtures:reset
```

The seed is idempotent. Re-running `npm run db:fixtures:seed` updates the stable
fixture records without multiplying rows or deleting unrelated records.

## Use The Fixture Database

Start the API in a PowerShell session configured for the fixture database and
isolated media root:

```powershell
$env:DATABASE_URL='postgresql://postgres:sa3421@127.0.0.1:5432/sidhkofed_seed?schema=public'
$env:STORAGE_PROVIDER='local'
$env:STORAGE_LOCAL_ROOT="$PWD\storage\seed-fixtures"
npm run dev
```

The Admin CMS can continue to use `http://localhost:3001`; its API base remains
`/api/v1`. The fixture script does not rewrite `.env`.

## Export, Restore, And Verify

```powershell
npm run db:fixtures:export
npm run db:fixtures:restore-verify
```

Exports are written under `artifacts/seed/`:

- `sidhkofed-seed.sql.gz` - portable PostgreSQL schema and data dump
- `sidhkofed-seed-media.tar.gz` - local media objects
- `manifest.json` - schema version, SHA-256 checksums, and exact row counts

Restore the committed artifacts back into the isolated fixture database:

```powershell
npm run db:fixtures:restore
```

Restore drops and recreates only `sidhkofed_seed` and replaces only
`storage/seed-fixtures`. `restore-verify` restores into a temporary
`sidhkofed_verify_seed`, compares every expected count and 40 media objects, then
removes the temporary database.

## Dataset Summary

| Area | Records |
|---|---:|
| Media assets | 40 |
| Galleries / gallery images / videos | 10 / 30 / 10 |
| Institutions / programmes / toolkits / toolkit items | 12 / 10 / 10 / 30 |
| Documents / events / news | 18 / 24 / 12 |
| Communications / tenders / procurement updates | 12 / 12 / 12 |
| Pages / menu items / FAQs / digital services | 12 / 24 / 12 / 10 |
| Memberships / enquiries | 20 / 15 |
| Dashboard reports / datasets / metrics | 13 / 13 / 52 |
| Event field definitions / audit logs / media usages | 12 / 36 / 46 |

The data includes public, draft, unpublished, private, future-scheduled, and
archived records; multiple event statuses; clean/suspected/spam enquiries;
bilingual and missing-translation cases; homepage flags; reporting periods; and
cross-module relationships.

## Schema Limitations

- `SuccessStory` has no implemented Prisma model or mounted route.
- Homepage data is aggregated from flagged module records; there is no homepage table.
- Search vectors are generated by PostgreSQL and are not directly seeded.
- Individual beneficiaries, inventory, procurement transactions, hosted videos,
  and configurable report builders are intentionally outside the schema.
- Fixed roles and canonical master catalogs legitimately contain fewer than 10 rows.
- MoUs are represented through Institution, Event, and Document relationships.
- Only the configured Super Admin is created; no additional test users are added.
